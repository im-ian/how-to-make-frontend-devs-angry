# 프론트엔드 개발자를 화나게 하는 방법

프론트엔드 개발자와 협업하게 되신 것을 축하드립니다.

이 문서는 프론트엔드 개발자와 더 자주, 더 긴밀하게, 더 격렬하게 대화하고 싶은 분들을 위한 실전 가이드입니다.

> 실제 협업에서는 따라 하지 마세요. 재미로만 읽으면 됩니다.

## 1. API는 아직인데 일정은 그대로 두세요

기획은 아직 확정되지 않았고, 디자인은 내일 나오고, API는 이번 주 안에 된다고 말하면 됩니다.

특히 백엔드 API는 완성하지 않은 채 매일 칼퇴하면 좋습니다. 왜냐고요? 재밌잖아요.

프론트엔드 개발자와 대화를 하고 싶다면 "일단 목데이터로 붙여주세요"라고 하면 바로 DM이 올 거예요.

## 2. 주석은 약속이 아니라 분위기입니다

타입에는 이렇게 써두면 됩니다.

```ts
amount: number // 수량 없으면 0
```

그리고 실제 응답은 이렇게 보내면 됩니다.

```json
{
  "amount": null
}
```

왜냐고요? 0과 null 사이에는 프론트엔드 개발자의 깊은 사색이 필요하니까요.

## 3. null이 싫다면 컬럼을 없애세요

스펙에는 이렇게 적으면 됩니다.

```ts
nickname: string | null
```

그리고 닉네임을 설정하지 않은 유저에게는 `nickname` 컬럼 자체를 보내지 않으면 됩니다.

프론트엔드 개발자와 대화를 하고 싶다면 "없는 값이라서 안 보냈어요"라고 하면 바로 DM이 올 거예요.

## 4. CORS는 프론트엔드 문제라고 말하세요

로컬에서는 안 되고, Postman에서는 되고, 브라우저에서만 터지면 이렇게 말하면 됩니다.

> 백엔드에서는 정상인데요?

왜냐고요? 브라우저가 막았으니 프론트엔드가 해결하면 될 것 같잖아요.

프론트엔드 개발자와 대화를 하고 싶다면 "프록시 설정하면 되지 않나요?"라고 하면 바로 DM이 올 거예요.

## 5. 숫자는 문자열로 보내세요

수량은 숫자처럼 보이면 충분합니다.

```json
{
  "amount": "10"
}
```

왜냐고요? 화면에는 어차피 10이라고 보이잖아요.

정렬, 계산, 검증은 프론트엔드 개발자가 알아서 하면 됩니다.

## 6. 컬럼명 규칙은 자유롭게 섞으세요

일관성은 지루합니다.

```json
{
  "nick_name": "ian",
  "nickName": "ian",
  "userId": 1,
  "userID": 1
}
```

스네이크 케이스, 카멜 케이스, 약어 대문자 규칙을 골고루 섞으면 됩니다.

왜냐고요? 자동완성도 가끔은 긴장해야 하니까요.

## 7. 날짜 포맷은 그날 기분에 맡기세요

어떤 API는 UTC로, 어떤 API는 ISO 문자열로, 어떤 API는 서버 로컬 타임으로 보내면 됩니다.

```json
{
  "createdAt": "2026-06-25T00:00:00Z",
  "updatedAt": "2026-06-25T09:00:00+09:00",
  "deletedAt": "2026.06.25 09:00:00"
}
```

프론트엔드 개발자와 대화를 하고 싶다면 "시간은 백엔드에서 맞게 내려가고 있어요"라고 하면 바로 DM이 올 거예요.

## 8. HTTP 메서드는 POST 하나면 충분합니다

조회도 POST, 수정도 POST, 삭제도 POST로 하면 됩니다.

```http
POST /api/user/get
POST /api/user/update
POST /api/user/delete
```

왜냐고요? 서버에서 분기하면 되잖아요.

PUT, PATCH, DELETE 같은 메서드는 프론트엔드 개발자가 REST 감성에 젖어 있을 때만 필요합니다.

## 9. 에러 응답은 성공처럼 보내세요

HTTP 상태 코드는 항상 200으로 통일하면 됩니다.

```json
{
  "success": false,
  "message": "권한이 없습니다."
}
```

왜냐고요? 네트워크는 성공했잖아요.

프론트엔드 개발자와 대화를 하고 싶다면 "status code는 원래 200으로 내려요"라고 하면 바로 DM이 올 거예요.

## 10. 문서는 배포 후에 업데이트하세요

API 스펙이 바뀌어도 문서는 그대로 두면 됩니다.

프론트엔드 개발자가 직접 콘솔을 찍고, 네트워크 탭을 열고, 응답을 비교하면서 최신 스펙을 발견하게 하면 됩니다.

왜냐고요? 탐험은 개발자의 기본 소양이니까요.
